- **Design notes:**
  - **`fs.File` is an `io.ReadCloser`:** `io/fs` declares `type File interface { Stat() (FileInfo, error); Read([]byte) (int, error); Close() error }`. Returning `fs.File` directly satisfies `io.ReadCloser`, so `Open` returns the underlying handle unchanged — callers close it when done.
  - **Peek is stateless (F4):** each call re-opens via `Open`, reads up to `n` via `io.ReadFull`, closes. No shared cursor on `*File`. `TestFile_Peek_MultipleCalls` asserts two calls return identical bytes; binary detection (3.4) and the Drop 4.1 shebang sniff both rely on this.
  - **`io.ReadFull` short-file semantics:** the stdlib doc is explicit — if no bytes were read the error is `io.EOF`; if some-but-fewer bytes were read the error is `io.ErrUnexpectedEOF`. Peek tolerates both (and `nil` for exact-match / long files) via `errors.Is`; any other error is wrapped with the shared `open %q: %w` prefix so callers can still extract the path and inspect the root cause. `errors.Is` over `==` so that a future stdlib wrapping change does not silently break this branch.
  - **`n <= 0` short-circuit:** `Peek(0)` and negative `n` values return `(nil, nil)` without opening the file. This avoids an ambiguous `make([]byte, n)` panic on negative input and keeps the "zero-byte peek" contract trivially correct.
  - **`Open` error wrapping:** `fmt.Errorf("open %q: %w", f.Path, err)` preserves the underlying `fs.PathError` chain that `fstest.MapFS` (and `os.DirFS`) return. `TestFile_Open_NotFound` asserts both `errors.Is(err, fs.ErrNotExist)` holds through the `%w` chain AND the text prefix is `open "missing.txt":` — the quote form matters because `%q` adds quotes around the path, giving a clearer error message when paths contain spaces.
  - **`IsHidden(".")` / `IsHidden("..")` → false** per C3. The function operates on a single basename (not a full path), so callers must pass `DirEntry.Name()` — the Walker in Unit 3.3 does exactly this. The doc comment calls this out so future callers don't feed full slash-separated paths and expect per-segment matching.
  - **Zero-value `File`:** exported fields + unexported `fs` field means a literal `fileset.File{}` compiles but `Open` / `Peek` will panic on the nil `fs.FS`. The doc comment explicitly flags zero-value as "not useful" and directs construction through `newFile`; the Walker (Unit 3.3) will be the only in-package constructor caller after the tests.
  - **Fixtures via `fstest.MapFS`** everywhere — zero disk IO, zero `testdata/` directory. This matches CLAUDE.md § "Tests" → "two-tier `testdata/` rule" (`internal/fileset/testdata/` is not created in this unit) and F11 (no binary fixtures in `internal/fileset/testdata/`).
  - **`t.Parallel()`** on every test function and every table subcase — the package holds no shared state and the MapFS instances are per-test, so parallel execution is safe and catches concurrency bugs that the race detector alone would not.

- **Design notes:**
  - **F14 yield-false guard (the load-bearing pin):** implemented as a captured `yieldOK` bool at the top of the `Walk` iterator closure. The `WalkDirFunc` checks it first thing on every invocation — if false, it immediately returns `fs.SkipAll` without touching yield. Every yield call-site flips `yieldOK = false` before returning `fs.SkipAll` when either (a) yield returns false, or (b) the branch is a "stop regardless" branch (ctx-cancel, invalid-matcher, internal walker error). The final yield is reached through a single code path per entry: hidden-check → depth-check → gitignore-read → matcher-check → yield. `TestWalker_RangeBreak` is the regression guard; it uses a 4-file MapFS (at-least-3-files per spec) and the test `defer recover()` catches any panic from a re-entered yield. Green.
  - **ctx.Done handling:** yielded once with `ctx.Err()` (context.Canceled or context.DeadlineExceeded), then `yieldOK = false` + `fs.SkipAll` so no further entries are visited. The caller's yield return value is discarded — the context-cancel contract trumps it.
  - **relPath normalization:** `relFrom(root, p)` returns `""` for the walk root itself (whether root is `"."` or any other path), matching the `ignore.GitignoreRoot.Dir` convention. Files under the root get their relative path with forward slashes (io/fs's native separator, forwarded verbatim; MapFS + os.DirFS both use forward slashes regardless of host OS). The initial `p == root` check covers both the `"."` root and the literal root-path match case; the fallback `strings.TrimPrefix(p, root+"/")` handles the "root is a non-`.` subdir" case. First bug fix of the round was in this function — the original guarded on `root == "." || root == ""` first, which returned `"."` for the walk root and broke the `relPath != ""` skip in the matcher-check step, dropping the entire walk when any include filter was set. Tests caught it immediately (`TestWalker_IncludeExclude` and `TestWalker_Gitignore` both failed with empty `paths`).
  - **Depth math:** counts forward slashes in `p` minus `slashCount(w.root)`. `"."` counts as zero slashes (root convention). Files at depth `N` live inside a directory at depth `N`; the directory prune happens via `fs.SkipDir` the first time WalkDirFunc visits a dir whose depth meets `>= Depth` — the walk root (depth 0) is explicitly exempt so `Depth=1` still walks root contents. File-level depth check is a defensive fallback that should not fire in practice because the containing directory's `SkipDir` already pruned the subtree. Depth `0` = unlimited per spec.
  - **Hidden entry handling:** calls `fileset.IsHidden(d.Name())` on the basename per C3. `Name()` from `fs.DirEntry` returns just the final element ("sub", "main.go", ".gitignore") — never a slash-separated path — so a single-basename `IsHidden` match is the correct semantic. The walk root is exempt (its `Name()` is `"."` from MapFS's fs.WalkDir, which `IsHidden` already treats as non-hidden). Hidden directories prune the subtree with `fs.SkipDir`; hidden files just return `nil`.
  - **Hierarchical gitignore (F8):** before descending into any directory, `readGitignore` opens `<dir>/.gitignore` via `fsys.Open`. Missing file (fs.ErrNotExist) / unreadable (permission) / empty-content cases all return nil — none are fatal. When a file exists and has lines, we append a fresh `ignore.GitignoreRoot{Dir: relPath, Patterns: lines}` to the roots slice and rebuild the `Matcher` via `ignore.New`. Rebuilding per-dir is O(N rules × M directories-with-gitignore), which is cheap relative to the walk itself (ignore.New just stores the pre-parsed slices; the gitignore library's `CompileIgnoreLines` does one regex compile per pattern, also linear). `TestWalker_NestedGitignore` asserts the F8 pin: a pattern in `sub/.gitignore` drops only paths under `sub/`, not sibling `secret.txt` at the root or `other/secret.txt` in a different subtree.
  - **Per-entry error yielding (F6):** `fs.WalkDir` reports ReadDir failures by calling WalkDirFunc a second time on the same directory path with `err != nil`. My closure yields that wrapped error (`fmt.Errorf("walk %q: %w", p, err)`) and returns `fs.SkipDir` so the walk continues. `TestWalker_UnreadableEntry` uses a custom `errFS` type whose `Open("broken")` returns an `errDir` stub that always fails ReadDir, and asserts the walk both yields the induced error AND continues to emit `keep.txt` / `other/y.txt`. Green.
  - **Symlink policy (F7):** I do NOT add any symlink-type check before yielding. `fs.WalkDir` does not follow symlinks (stdlib-documented), so a symlink encountered in-tree is reported as a plain entry via `fs.DirEntry.Type() & fs.ModeSymlink != 0`. My walker yields it as a regular file; `File.Open` on a broken symlink returns `open "link_broken": lstat missing.txt: no such file or directory` wrapping `fs.ErrNotExist`. `TestWalker_SymlinkYielded` asserts both link_ok and link_broken are yielded, and that `broken.Open()` unwraps to `fs.ErrNotExist` via `errors.Is`. Green.
  - **Context cancel test:** uses `context.WithCancel` + `cancel()` before iterating. The first WalkDirFunc invocation (on root `"."`) sees `ctx.Err() != nil`, yields `context.Canceled`, flips guard, returns `fs.SkipAll`. The test asserts zero paths + at least one yielded error unwrapping to `context.Canceled`. Green.
  - **`errFS` test stub design:** implements only `fs.FS.Open`, delegates to inner `fstest.MapFS` for every path except the configured `failDir`. For the failing directory, returns an `errDir` that implements both `fs.File` and `fs.ReadDirFile` (the latter via `ReadDir(n int) ([]fs.DirEntry, error)` returning an induced error). This is the minimum surface needed to exercise `fs.WalkDir`'s "second call with err" branch. No attempt to model fs.StatFS / fs.GlobFS / fs.ReadDirFS — the walker's only required method on fsys is Open.
  - **F12 (CLI-free):** `internal/fileset` imports only `bufio`, `bytes`, `context`, `fmt`, `io/fs`, `iter`, `path`, `strings`, and the sibling `internal/ignore`. No cobra, no spf13/pflag, no laslig, no flag parsing. All CLI policy (`--binary`, `--no-gitignore`, `--include`, `--exclude`, `--depth`, `--hidden`) belongs to Unit 3.5's wire-up; this unit exposes fields on `WalkOptions` that the CLI layer will populate.
  - **`iter.Seq2` adoption (F5):** single-producer range-over-func — `for f, err := range w.Walk(ctx)`. The returned `iter.Seq2[*File, error]` closure is called once per iteration by the Go runtime, and break / early return halt cleanly via the F14 yield-false pathway. No channels. `Walk` can be called multiple times on the same `*Walker` (each call returns a fresh closure); the test suite exploits this in `TestWalker_SymlinkYielded` by iterating twice over the same walker to locate the `link_broken` file.
  - **Library usage:** `ignore.New` with `nil` excludes / includes / roots produces a matcher that accepts everything — the first Walk iteration on an empty tree builds this trivial matcher and never returns an error (glob validation is the only error path and empty slices pass trivially). The `matcherErr` cache stores the construction error in case Unit 3.5 ever passes invalid glob patterns; in that case the walker yields one `walker: invalid glob pattern ...` error and stops. This path is covered indirectly by Unit 3.1's `TestMatcher_InvalidGlob_*` and is defensive-only here — no test constructs a walker with invalid patterns because Unit 3.5 hasn't landed.
  - **`t.Parallel()`** on every test function and every table subcase — matches Unit 3.1 / 3.2 style. Race detector on by default via `mage test`.
  - **Test helper design:** `collect(t, w)` / `collectCtx(t, w, ctx)` drain the walk into `walkResult{paths, errs}`. Two helpers instead of one because the default-context case is the common path and the ctx-cancel case is one test; separate functions avoid needing the full context parameter at every call site.
  - **`slices.Sort` for deterministic assertions:** `fs.WalkDir` visits entries in lexical order, so emission order is deterministic — but I sort both got and want before comparing because MapFS's ordering is not specified by the fstest docs in every version, and the tests should not regress if the stdlib tightens or relaxes ordering guarantees.

- **Design notes:**
  - **F9 sentinel shape:** `ErrBinaryFile = errors.New("binary file")` per `CLAUDE.md § "Errors"` and the planner's explicit direction — inspected via `errors.Is`, never string-matched. `IsBinary` itself does NOT return `ErrBinaryFile`; it reports the classification via its bool. The sentinel exists for callers (aggregation loop in Unit 3.5's `cmd/rak/root.go`) that translate "IsBinary returned true → skip this file" into an error value for the render error summary. Doc comment on `ErrBinaryFile` spells this out so future readers don't wonder why the sentinel is unused within the package.
  - **F10 heuristic:** single `bytes.IndexByte(peek, 0x00) >= 0` scan over whatever `Peek(512)` returns. No `strings.IndexByte` fork because `bytes.IndexByte` operates directly on the `[]byte` returned by `Peek` (one cast avoided). The "NUL past byte 512" regression guard (`nul_past_peek_window_is_not_binary`) uses a 521-byte fixture with NUL at index 520; `Peek(512)` returns the first 512 'A' bytes, and `IndexByte` finds no NUL. If the implementation ever grew the peek window past 512 (or forgot the window bound), this test flips to `want: true` and fails — the exact F10 pin the planner asked for.
  - **Empty file → not binary:** explicit `len(peek) == 0` guard before the NUL scan. `bytes.IndexByte` on an empty slice returns `-1`, so the scan itself would produce the same answer, but the explicit guard makes the semantic intent obvious in the code and matches the planner's acceptance wording ("Empty file → not binary (len(peek) == 0 → false)").
  - **`bytes` vs `strings`:** used `bytes.IndexByte` because `Peek` returns `[]byte`. The planner's acceptance permits either (`bytes.IndexByte(peek, 0x00) >= 0` or `strings.IndexByte` over byte slice); `bytes.IndexByte` wins because it takes `[]byte` directly — no string conversion, no allocation. Both functions dispatch to the same SIMD-accelerated runtime primitive on supported architectures, so performance parity is irrelevant.
  - **`IsBinary` error propagation (C10):** the function has exactly one error source — `f.Peek(512)`. The NUL scan is a pure byte-slice operation that cannot fail. No wrapping at this layer because `Peek` already wraps with `open %q: %w` from Unit 3.2. Additional wrapping would be noise.
  - **No Walker wiring (F12):** `binary.go` adds `IsBinary` as a method on `*File` only. Nothing in `walker.go` touches it; no CLI-coupling. The Walker yields every non-ignored file; the decision to skip binaries belongs to `cmd/rak` in Unit 3.5. Verified by grep: no `IsBinary` / `ErrBinaryFile` references in `walker.go`, no cobra / flag imports added to the package.
  - **No binary fixtures on disk (F11):** every test case builds content inline via `[]byte` literals (`[]byte("hello world")`, `[]byte{0x00, 0x01, 0x02, 0x03}`, PNG magic bytes as a `[]byte{0x89, 'P', 'N', 'G', ...}` literal) or via the `buildASCII` / `buildASCIIThenNULAt` helpers. Zero `testdata/` growth. `internal/fileset/testdata/` remains non-existent, per the two-tier testdata rule.
  - **`t.Parallel()`** on the top-level `TestFile_IsBinary` and every subcase — matches Unit 3.1 / 3.2 / 3.3 style. Each subcase builds its own `fstest.MapFS` so there is no shared state to race on.
  - **Helper visibility:** `buildASCII` and `buildASCIIThenNULAt` are unexported package-level helpers in `binary_test.go` (_test.go files live in the same package here — `package fileset`, not `package fileset_test`). Keeping them `*_test.go`-scoped means they don't leak into the binary-release surface and aren't visible to other packages. Doc comments cover intent and bounds contract.
  - **Package doc already covers the file:** `file.go`'s package comment mentions that "binary detection (Unit 3.4) and shebang sniff (Drop 4.1) can call [Peek] independently." No new package comment in `binary.go`; per Go convention a package comment lives on exactly one file (`file.go` owns it here).
  - **gofumpt + lint pass on first try:** no post-hoc reformatting needed. The `defer rc.Close()` errcheck gotcha from Unit 3.2 doesn't recur because `IsBinary` does no file I/O directly — `Peek` already handles Close and error wrapping.
