# Drop 4 — Plan QA Proof, Round 1

**Verdict:** pass-with-findings

## Summary

The plan is structurally sound — six atomic units with a clean build graph (4.0 → {4.1 → 4.2 → 4.3, 4.5} → 4.4), preserved Drop 3 contracts (F25), and well-reasoned F-pins (F16–F25) that cover the load-bearing invariants. External API claims about cobra (`MarkFlagsMutuallyExclusive`) and toon-go (`Marshal`, option functions, struct tags) are verified against Context7 docs.

However, multiple findings warrant fixes before build. The most consequential is **P1**: the planner's git-ls-files invocation strategy ("run from `g.toplevel`, then filter by prefix") is built on a false premise — `git ls-files --full-name` invoked with `cmd.Dir = walk-root-subdir` already scopes to the subdir AND emits toplevel-relative paths, so the entire prefix-filter loop is dead code. **P2** flags the 4.1 forward-declaration trick (package-level `var newGitLister/newWalkLister`) as needing tighter spec to compile cleanly. Smaller findings cover the `Detect` non-disk-FS contract gap (P4), the `dirKey` per-dir rollup invariant missing from F-pins (P5), TOON `omitempty` not actually documented in the toon-go API (P6), an inconsistency in 4.4 about `runDirectory`'s `fsys` parameter (P7), and a coverage gap on `fs.FS` injection for tests (P8).

## Findings

### Finding P1 — GitLister.List `cmd.Dir = g.toplevel` + prefix-strip strategy contradicts actual `git ls-files` scoping

- **Severity:** blocker
- **Unit affected:** 4.2 (Acceptance bullets for `GitLister.List`), F17
- **Claim:** The plan says `(g *GitLister) List` runs `exec.CommandContext(ctx, "git", "ls-files", "--full-name", "-z")` with `cmd.Dir = g.toplevel` and then iterates the full repo's tracked files, **filtering** out paths that don't have `g.prefix` as a prefix. The prefix-skip and prefix-strip logic in steps (1)+(2) of 4.2 is the load-bearing path for sub-directory walks (e.g. `rak ./internal/fileset/`).
- **Evidence missing or weak:** Empirically verified against the rak repo: `git -C /Users/evanschultz/Documents/Code/hylla/rak/main/internal/fileset ls-files --full-name` returns **only the 6 files inside that subdirectory**, NOT all 53 tracked files in the repo (Bash tool invocation, this session). `git help ls-files` confirms the design: `--full-name` "When run from a subdirectory, the command usually outputs paths relative to the current directory. This option forces paths to be output relative to the project top directory." That is — `git ls-files` always scopes by CWD; `--full-name` only changes the path representation in the output. So running with `cmd.Dir = g.toplevel` enumerates **all repo files** (which then need filtering), but running with `cmd.Dir = absRoot` enumerates **only the subdir** (no filter needed). The plan picks the more expensive + more error-prone option.
- **Suggested fix:** Either (a) run `git ls-files --full-name -z` with `cmd.Dir = g.root` (the absolute walk root) — output already scoped to the subdir, paths are toplevel-relative, you still strip `g.prefix` to get walk-root-relative `relPath`. The skip-by-prefix step becomes unnecessary. Or (b) keep `cmd.Dir = g.toplevel` but document the rationale (e.g. "consistency, single CWD for all git calls") and accept the filter cost. The plan's current text combines (b)'s implementation with what reads like (a)'s rationale ("paths that have prefix as a prefix" implies most won't, which is only true if we read the whole repo unnecessarily). Pick one path explicitly. Either way, update F17 to match.

### Finding P2 — 4.1 "forward-declare newGitLister / newWalkLister" approach under-specified, risks compile failure between units

- **Severity:** major
- **Unit affected:** 4.1 (Acceptance, the "forward-declared stub" paragraph)
- **Claim:** The plan says builder may forward-declare `newGitLister` / `newWalkLister` as package-level `var`s with nil defaults to be replaced by real constructors in 4.2 / 4.3. The stated benefit: `mage build` passes after 4.1 lands without 4.2/4.3.
- **Evidence missing or weak:** A package-level `var newGitLister = func(...) (*GitLister, error) { ... }` declaration requires the `*GitLister` type to exist at 4.1's commit boundary — but 4.2 owns `git.go` which is where `type GitLister` lives. Either (i) `type GitLister struct{}` shell types ALSO get forward-declared in 4.1's lister.go (then 4.2 re-declares = compile error, OR 4.2 promotes the shell, which is murky surgery), or (ii) the var has `func(...) (FileLister, error)` return type (interface), which avoids the type circularity but means `Detect` cannot type-assert anywhere — and 4.1's tests assert on `*GitLister` concrete type. The plan does not resolve this. The fallback ("`Detect` as a thin dispatch that calls constructors defined in 4.2/4.3, and accept that `mage build` of the package requires all three units done") gives up the per-unit-buildable property entirely, which is the very thing the plan-unit decomposition is meant to enforce.
- **Suggested fix:** Pick **one** strategy explicitly. Recommendation: collapse 4.1 + 4.2 + 4.3 into a single "lister package" unit, OR move 4.1's `Detect` body to 4.4 (so 4.1 ships just the `FileLister` interface + empty package) and have 4.2 own `newGitLister` + the dispatch logic that produces it, 4.3 own `newWalkLister`, 4.4 own `Detect` which now sees both constructors. The current text reads "builder picks the cleanest option" — which is a punt that QA will catch as a finding twice (once in proof, once in falsification). Resolve at plan time.

### Finding P3 — 4.1 `TestDetect_InsideRepo` test against actual checkout couples unit tests to local git state

- **Severity:** minor
- **Unit affected:** 4.1 (Acceptance, test bullets)
- **Claim:** `TestDetect_InsideRepo` runs against the actual checkout directory (`main/`) using `os.DirFS`; verify returned type is `*GitLister`. `TestDetect_OutsideRepo` uses `t.TempDir()` and expects `*WalkLister`.
- **Evidence missing or weak:** "actual checkout" tests run fine locally and in CI from the cloned repo, but anyone running `go test ./internal/lister` from a tarball, a Bazel/Pants sandbox, or a `go install`-pulled vendor tree without `.git/` present silently flips the test path (no git = `*WalkLister`, contradicts the assertion). The plan does note `t.Skip` if `git` binary absent, but does not address "git binary present, .git/ absent" — the test will fail there, not skip. Also `TestDetect_OutsideRepo` should worry that `t.TempDir()` on macOS lives under `/var/folders/...` — outside any git repo by default — so the assertion holds; but if the CI runner is itself inside a checked-out git tree and `t.TempDir()` happens to resolve to a path inside, the test could flip. Probably fine in practice, but worth a defensive `git rev-parse --is-inside-work-tree` check via shell first.
- **Suggested fix:** Replace `TestDetect_InsideRepo` with a setup that **creates** an empty git repo in `t.TempDir()` via `exec.Command("git", "init", tmpDir)` then asserts. That makes the test hermetic — no dependence on the developer's checkout layout — and gives a true positive control. Skip the entire test pair if `git` binary is absent. Apply the same hermetic pattern to GitLister's tests in 4.2 (currently they walk the live checkout, see P9).

### Finding P4 — `Detect` and `GitLister` undefined behavior for non-disk `fs.FS`

- **Severity:** major
- **Unit affected:** 4.1 (`Detect` signature), 4.4 (`runRoot` call site), F-pin gap
- **Claim:** `Detect(ctx, root, fsys, opts)` takes an `fs.FS` parameter that gets handed down into `WalkLister` (via `fileset.NewWalker(fsys, root, opts)`). But `Detect` itself runs `exec.Command("git", "rev-parse", "--is-inside-work-tree")` with `cmd.Dir = root` — which is a real disk path, not an `fs.FS` operation. So `Detect` will happily ignore the `fsys` parameter when probing for git, then either dispatch to `GitLister` (which also ignores `fsys` because it shells `git ls-files`) or `WalkLister` (which **does** use `fsys`).
- **Evidence missing or weak:** This is an inconsistency: when a test injects an `fstest.MapFS` as `fsys` AND root is a real disk path inside a git repo, `Detect` will dispatch to `GitLister`, which then reads files from disk via `git ls-files`, but `newFile(g.fsys, relPath, relPath)` constructs a `*fileset.File` whose `f.fs == g.fsys` is the MapFS — so subsequent `f.Open()` calls go to the MapFS, not disk. Result: the file list is real, the file contents are fake. Confusing. The plan doesn't pin which `fs.FS` value `GitLister.List` should use for `newFile`. There are three options: (a) always `os.DirFS(g.toplevel)`, ignore the caller-passed `fsys`; (b) use the caller-passed `fsys` and document that GitLister is "disk-backed" so injecting a fake `fs.FS` is undefined behavior; (c) emit a runtime error if `Detect` sees a non-`*os.dirFS` and a git repo is present.
- **Suggested fix:** Add a new F-pin (F26) that nails down the disk-vs-FS contract: GitLister always uses `os.DirFS(g.toplevel)` for newFile; if a caller wants to test git semantics, they need a real on-disk repo (matches P3). The `fsys` parameter to `Detect` is then only used by `WalkLister`. The 4.4 wire-up should call `lister.Detect(ctx, args[0], nil, opts)` from production code (let Detect resolve `os.DirFS(args[0])` internally for the WalkLister branch), OR `lister.Detect` should not take `fsys` at all — only the walker-only constructor (e.g. `lister.NewWalkLister(fsys, root, opts)`) takes one, used directly by tests. Either is fine but pick one.

### Finding P5 — 4.4 changes break `dirKey` / `labelDirectories` semantics for nested GitLister output; no F-pin captures this

- **Severity:** major
- **Unit affected:** 4.4 (Acceptance, integration tests), Notes "Integration Test Impact", F-pin gap
- **Claim:** The current `walkAndCount` calls `fileset.NewWalker(fsys, ".", ...)` — root is always `"."`, so `dirKey("sub/nested.txt") = "sub"` and `labelDirectories` rewrites `"."` → `rootLabel`, `"sub"` → `rootLabel + "/sub"`. After 4.4 switches to `lister.Detect(ctx, args[0], os.DirFS(args[0]), ...)`, the GitLister branch produces `*fileset.File` with `RelPath = "a.txt"` (because the plan strips `prefix + "/"`). So far so good — but in P1 above, when the user runs `rak ./internal/fileset/`, `g.prefix = "internal/fileset"` and the GitLister emits `RelPath = "file.go"`, etc. (already root-relative). Then `dirKey("file.go") = "."` (good) and `labelDirectories(".", rootLabel)` rewrites to `rootLabel = "./internal/fileset"` — output is `dir: ./internal/fileset`. That looks correct.
- **Evidence missing or weak:** But what about the integration test (Notes section "Integration Test Impact on testdata/tree")? `args[0] = "testdata/tree"`. After Detect, GitLister sees `cmd.Dir = /abs/path/to/cmd/rak/testdata/tree`, `toplevel = /abs/path/to/rak/main`, `prefix = "cmd/rak/testdata/tree"`. `git ls-files --full-name -z` (from toplevel per P1) returns paths like `cmd/rak/testdata/tree/a.txt`, `cmd/rak/testdata/tree/sub/nested.txt`. After prefix-strip, `relPath = "a.txt"`, `"sub/nested.txt"`. `dirKey("a.txt") = "."`, `dirKey("sub/nested.txt") = "sub"`. `labelDirectories` rewrites to `"testdata/tree"` and `"testdata/tree/sub"`. **All correct** — but this analysis is not in the plan; the plan just says "verify integration tests still produce correct counts." Specifically, F-pin F17 mentions the prefix computation but says nothing about ensuring `RelPath` stays compatible with `dirKey`. F23 says walkAndCount's binary check is preserved verbatim, but doesn't pin the contract that `*fileset.File.RelPath` from a `FileLister` source is always walk-root-relative (forward-slash, no leading `./`).
- **Suggested fix:** Add F26 (or whatever number after the F25 gap): "`FileLister.List` emits `*fileset.File` with `RelPath` walk-root-relative, forward-slash, no leading `./` — same shape `fileset.Walker` produces. `dirKey` (in `cmd/rak/root.go`) and `labelDirectories` operate on this shape; both `GitLister` and `WalkLister` must preserve it." This is the load-bearing invariant for per-dir rollups and is currently implicit.

### Finding P6 — `omitempty` claim in 4.5 not backed by Context7 evidence

- **Severity:** minor
- **Unit affected:** 4.5 (Acceptance, `RenderTree` bullet)
- **Claim:** "Emit `errors` only when non-empty (use `omitempty` equivalent — if toon-go supports `toon:",omitempty"` tag, use it; otherwise conditionally include the field)."
- **Evidence missing or weak:** Context7 `/toon-format/toon-go` query for "omitempty optional field" returned no docs mentioning `,omitempty`. The library's documented option functions are `WithLengthMarkers`, `WithArrayDelimiter`, `WithDocumentDelimiter`, `WithIndent`, `WithStrictMode`, `WithDecoderIndent` — no field-level omit-empty option surfaced. Comparing with stdlib `encoding/json`'s well-known `omitempty` is plausible but not verified for toon-go. The plan hedges ("if toon-go supports... otherwise conditionally include"), but the hedge bleeds into the implementation strategy and `rendere_test.go` expectations (`TestTOONRenderer_RenderTree_NoErrors`).
- **Suggested fix:** Drop the speculative `,omitempty` path. Bake in the conditional-construction approach: builder defines `toonTree struct { Directories []toonDir; Total toonCounts; Errors []string }` and constructs a separate struct without the `Errors` field (or assigns `nil` and asserts the lib omits nil slices — also test-and-see, not spec-backed) when the input `errs` is empty. Cite the actual lib behavior in `BUILDER_WORKLOG.md` when the unit lands. F20 is fine as-is; add a builder-side note in 4.5 that confirms the chosen approach.

### Finding P7 — 4.4 `walkAndCount` signature change has an extra inconsistency about `fsys`

- **Severity:** minor
- **Unit affected:** 4.4 (Acceptance, `walkAndCount` bullet)
- **Claim:** "`walkAndCount` signature: accept `FileLister` (not `fs.FS` + `flags`). Iterate via `lister.List(ctx)`. ... The `fsys` parameter is dropped (lister owns the fs.FS now). The `flags.binary` check is preserved — `walkAndCount` still receives enough information to apply the binary-skip policy; thread `binary bool` as a parameter."
- **Evidence missing or weak:** Current `walkAndCount(ctx, fsys, flags)` signature uses `flags.binary`, `flags.depth`, `flags.hidden`, `flags.noGitignore`, `flags.includes`, `flags.excludes` — six fields. The plan says drop `fsys`, drop `flags` (replace with `binary bool`). But the depth/hidden/no-gitignore/includes/excludes fields are already consumed by `lister.Detect`-side `WalkOptions` (great), so they don't need to thread into walkAndCount. Only `binary` is needed downstream. OK — the plan IS consistent on this. The wording is just confusing: "lister owns the fs.FS now" reads like the FileLister has a public fs.FS, but it does not (it's an unexported field in GitLister / a Walker-wrapping field in WalkLister). Reword to: "The `fsys` parameter is dropped from `walkAndCount` because the FileLister implementation already holds the fs.FS internally and exposes only `List(ctx)`."
- **Suggested fix:** Re-word as above. No functional change.

### Finding P8 — runTreeFS helper's path to constructing a test `FileLister` is ambiguous; risks builder coupling tests to git binary

- **Severity:** minor
- **Unit affected:** 4.4 (Acceptance, `runTreeFS` update bullet)
- **Claim:** "Update `runTreeFS` helper: ... The lister for those tests: construct a `lister.WalkLister` via `lister.NewWalkLister` (or a package-internal helper) with the injected `fstest.MapFS`. Alternatively, keep `runTreeFS` calling `runDirectory` with a stub that bypasses `lister.Detect` entirely — pass a pre-built `WalkLister` directly. Builder decides cleanest form."
- **Evidence missing or weak:** The current `TestRootCmd_PathArg_*` tests (root_test.go lines 148–390) inject `fstest.MapFS` to exercise gitignore + depth + binary + hidden paths. Those test invariants must keep working in 4.4 without requiring a real git repo on the test host. The plan correctly observes that we need a way to bypass `Detect` for those tests, but punts on the choice: an exported `lister.NewWalkLister` constructor breaks the package's encapsulation (it leaks `WalkLister`'s constructor); a package-internal helper requires the test to be in the `lister` package (or a `lister_internal_test.go`); passing a "pre-built WalkLister directly" requires `runDirectory` to accept a `FileLister` (which the plan does require) but doesn't pin the test plumbing.
- **Suggested fix:** Pin the choice. Recommendation: export `NewWalkLister(fsys, root, opts)` from `internal/lister` (it's an `internal/` package — no public API risk). Tests then do `lister.NewWalkLister(fstest.MapFS{...}, ".", fileset.WalkOptions{...})` directly. The `runTreeFS` helper becomes: build the lister, call `runDirectory(ctx, &out, "", lister, renderer)`. No call to `Detect`. Encode this in 4.4's acceptance bullet so the builder doesn't have to re-decide.

### Finding P9 — 4.2 GitLister tests rely on `filepath.Abs("../../..")` from `internal/lister/` to reach `main/` — brittle and unstated convention

- **Severity:** minor
- **Unit affected:** 4.2 (Acceptance, `TestGitLister_List_InRepo` bullet)
- **Claim:** "Use `filepath.Abs("../../..")` to reach `main/` from `main/internal/lister/`."
- **Evidence missing or weak:** Test packages run with CWD = the package directory by default (per `go help test`'s implicit fixture-locality, and the established pattern with `cmd/rak/testdata/`). `../../..` from `main/internal/lister/` is indeed `main/`. But this depends on the test never being moved, never being vendored under a different layout, never being copied. The brittleness is small but real. Worse: it tightly couples the unit test to the rak repo's directory layout, which means **if anyone runs the test from `main/`** (via `go test ./internal/lister/...`), CWD is still `main/internal/lister/` (Go convention), so `../../..` resolves correctly. OK in production. But: if the test asserts "the output includes go.mod" (the plan says "verify the list includes at least `go.mod`"), it relies on the rak repo always having `main/go.mod` tracked — which it always will, but again a fragile coupling.
- **Suggested fix:** Follow P3's recommendation: spin up an empty git repo in `t.TempDir()`, add a known file, commit, then assert. Hermetic, no parent-traversal magic, no coupling to rak's checkout layout. Skip on missing `git` binary as the plan already specifies. (This pairs with P3 — both findings call for the same fix.)

### Finding P10 — F19 (`--no-gitignore` no-op for GitLister) lacks `--help`-text rationale, blocked on Open Unknown 1 but not pinned for QA falsification

- **Severity:** minor
- **Unit affected:** Notes, Open Unknowns #1, F19
- **Claim:** F19: "`git ls-files` only lists tracked files, which are already gitignore-filtered by git. The `DisableGitignore` field in `WalkOptions` has no effect on `GitLister.List`. This semantic difference is intentional and documented here; no user-visible behavior change is needed in v0.1.0." Open Unknown 1: "Should rak document this in `--help` output, or silently ignore the flag?"
- **Evidence missing or weak:** F19 is an F-pin (invariant), but the question of `--help`-text documentation is parked in Open Unknowns for dev discussion. QA falsification will not be able to test "does `--no-gitignore` actually do nothing in a git repo?" against the F-pin alone if the test fixture is also git-tracked — `git ls-files` will skip the gitignored file regardless of flag, so the only way to falsify a "no-op" claim is to test against an integration fixture where git has been told to track an otherwise-gitignored file (via `git add -f`). That fixture doesn't exist in cmd/rak/testdata/tree, and the plan doesn't direct 4.4's integration test additions to create one. The pin and the test surface are not aligned.
- **Suggested fix:** Either (a) add a dedicated regression test in 4.2 that creates a tmp git repo, force-adds a file matching the repo's `.gitignore`, then runs `GitLister.List` with `DisableGitignore: true` and `false`, asserting both yield the same set; or (b) downgrade F19 from "F-pin invariant" to "Note: documented limitation, no test coverage in Drop 4" and update the F-pin language to match. F-pins should be testable; F19 currently sits in an awkward middle ground.

### Finding P11 — Default Detect failure mode (exit != 0 from `rev-parse`) silently dispatches to WalkLister; not explicitly documented and may mask real errors

- **Severity:** minor
- **Unit affected:** 4.1 (`Detect` bullet)
- **Claim:** "On exit code 0, returns a `*GitLister`. On any other exit (not in git repo, git binary absent), returns a `*WalkLister` (built by `newWalkLister`)."
- **Evidence missing or weak:** `git rev-parse --is-inside-work-tree` exits non-zero for at least three distinct conditions: (1) not inside a git repo (correctly handled by falling back to WalkLister), (2) `git` binary absent (`exec.Command` returns an error before the process exits — handled by the `os.exec` API), (3) git is present but `--is-inside-work-tree` fails for some non-zero/non-"not in repo" reason (e.g. broken `.git/` state, corrupt index). Case (3) silently degrades to WalkLister with no surfacing of the underlying git error. The plan says "Wraps errors with context" but the dispatch logic loses the error. For a CLI tool, silently falling back is the right product call, but the plan should pin it as intentional (an F-pin) so QA falsification doesn't flag it as a bug.
- **Suggested fix:** Add F-pin (F27 or similar): "`Detect` treats any non-zero exit from `git rev-parse --is-inside-work-tree` AND any `exec.Command` error (including `git` binary absent) as 'not inside a git repo' — dispatch to `WalkLister`. Underlying git errors are not surfaced. Rationale: rak is a counting tool, and falling back to filesystem walk is always a safe default when git is unavailable for any reason." This gives QA falsification a target to attack ("but what if `git rev-parse` exits 128 because the index is corrupt?") and the answer is encoded in the pin.

### Finding P12 — `t.Skip` on absent git binary good, but no plan-level test for "git crashes mid-walk"

- **Severity:** nit
- **Unit affected:** 4.2 (Acceptance, missing test case)
- **Claim:** Plan covers `TestGitLister_ContextCancel` (cancel context, first yield is `(nil, ctx.Err())`) but no case for `git ls-files` exiting with a non-zero status during the read (e.g. permission denied on an object, partial buffer).
- **Evidence missing or weak:** `exec.Command(...).Output()` returns stdout + a non-nil error if `git` exited non-zero. The plan doesn't pin what `GitLister.List` does in that case. Should it yield `(nil, wrappedErr)` then stop, or yield successful entries first then the error? In practice `git ls-files` either succeeds fully or fails before producing output, so the question may be moot — but absence of a test means the iterator's error-path semantics are unverified.
- **Suggested fix:** Add a test case `TestGitLister_GitExitError` that stubs `exec.Command` (or uses a wrapper interface so the test can inject a fake git command that exits non-zero). Skip the test if the stubbing is too invasive — but at least pin the expected behavior in F-pin form: "If `git ls-files` exits non-zero, GitLister yields `(nil, wrappedErr)` once and terminates." This is a low-priority finding; ignoring it is reasonable if scope is tight.
