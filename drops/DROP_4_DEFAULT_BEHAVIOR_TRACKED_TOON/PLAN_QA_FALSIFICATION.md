# Drop 4 — Plan QA Falsification, Round 1

**Verdict:** fail

## Summary

Fifteen counterexamples land. Six are blockers: the integration fixture's `vendor/` tree is **gitignored, not tracked**, so the existing `TestRootCmd_PathArg_Gitignore` semantics cannot survive 4.4 unchanged (C1); 4.4 silently drops the existing per-fs.FS-injection test helper `runTreeFS` without replacing the contract, leaving five Drop 3 tests orphaned (C2); `git ls-files --full-name -z` paths use forward slashes on every host while `F17` strips `toplevel+string(filepath.Separator)` and would fail to strip on Windows (C3); `F18`'s "hidden check per segment" calls `fileset.IsHidden(name)` which takes a single basename, not a slash-joined path — no per-segment loop is specified (C4); `F19`'s `--no-gitignore` no-op contradicts decision 32 and decision 10's user-trust contract (C5); the integration test's `runDirectory` call goes through `lister.Detect` and **the test's working directory is `main/cmd/rak`** — inside the rak git repo — so `Detect` returns a `GitLister` that enumerates from the repo toplevel, not from the `os.DirFS(args[0])` the user expects (C6). The remaining nine are major or minor — toon-go `omitempty` is unverified; pipe-in-value escaping is unverified; submodule + sparse-checkout + worktree edge cases are not specified; `F17` empty-prefix case has an off-by-one; `mage build` semantics for 4.1 alone are still ambiguous; UN3's `.gitignore`-is-hidden carve-out contradicts decision 34 (count what git tracks).

## Counterexamples

### Counterexample C1 — Integration fixture `vendor/ignored.txt` is gitignored, not tracked; GitLister will NOT see it

- **Severity:** blocker
- **Attack target:** Unit 4.4 acceptance "All integration tests that use a directory path arg now go through `lister.Detect`, which will return a `GitLister` for the `cmd/rak/testdata/tree` path" + Notes § "Integration Test Impact on testdata/tree"

The plan's Notes paragraph asserts "the integration test's expected counts" still match Drop 3 — but specifically, Drop 3's `TestRootCmd_Integration_PathArg_JSONFormat` asserts `len(parsed.Directories) != 2` (root + sub, NOT vendor) and the totals **already exclude `vendor/`**. The plan implies this is fine. It IS fine for the path tests. But the plan does **NOT** address what happens to `cmd/rak/root_test.go`'s `TestRootCmd_PathArg_Gitignore.no_gitignore_includes_vendor` (lines 221–233 of `root_test.go`): that test passes `noGitignore: true` and asserts `vendor/` is **counted**. The plan should pin: which lister does the path-arg test path use? If `GitLister`, then drop the `no_gitignore_includes_vendor` semantic from the integration coverage and document the loss; if `WalkLister`, then `runTreeFS` must hard-wire a `WalkLister` and the integration test's CWD-inside-git-repo problem (C6) is also resolved.
- **Mitigation status:** UNMITIGATED — plan must change.
- **Suggested fix:** Add to Unit 4.4 acceptance: "`runTreeFS` constructs `lister.NewWalkLister(fsys, ".", listerOpts(flags))` directly (bypasses `lister.Detect`) so injected `fstest.MapFS` tests preserve their existing semantics including the `no_gitignore_includes_vendor` subtest." OR explicitly drop the subtest with a Notes paragraph acknowledging the regression. Same for `TestRootCmd_PathArg_Gitignore.default_drops_vendor` — that still works because gitignore is applied by both paths, but call it out.

### Counterexample C3 — F17 uses `string(filepath.Separator)` for path-separator stripping but `git ls-files --full-name` always emits forward slashes

- **Severity:** blocker (Windows; minor for Unix-only target)
- **Attack target:** F17 — "`prefix = filepath.ToSlash(strings.TrimPrefix(absRoot, toplevel+string(filepath.Separator)))`"
- **Construction:** On Windows, `filepath.Separator == '\\'`. `absRoot = "C:\\Users\\foo\\repo\\cmd"` and `toplevel = "C:\\Users\\foo\\repo"` (from `git rev-parse --show-toplevel`, which on Windows returns FORWARD slashes by default — git outputs `C:/Users/foo/repo`). So `toplevel+string(filepath.Separator)` = `"C:/Users/foo/repo\\"` which is NOT a prefix of `absRoot` (Windows-native backslash). `TrimPrefix` returns the full string unchanged. Then `filepath.ToSlash` converts the backslashes. The resulting `prefix` is the FULL absolute path of `absRoot` with forward slashes — completely wrong. The downstream filter `strings.HasPrefix(repoRelPath, prefix)` never matches and GitLister yields zero files. Even on macOS, `filepath.Separator == '/'` so the test fixture might pass by accident, hiding the latent Windows bug. The plan should either pin "rak is Unix-only in v0.1.0; document Windows-unsupported in the README" OR normalize both sides to forward-slash before the TrimPrefix (`prefix = strings.TrimPrefix(filepath.ToSlash(absRoot), filepath.ToSlash(toplevel)+"/")`). The integration test (line 233) already includes a comment "rak's target platforms are Unix-like" — so the cleanest fix is to pin Unix-only and add a build tag, but the plan should make that pin explicit.
- **Mitigation status:** UNMITIGATED — plan must change.
- **Suggested fix:** Reword F17 to: "Normalize both `absRoot` and `toplevel` to forward slashes via `filepath.ToSlash` BEFORE the `TrimPrefix`, then strip a forward-slash separator: `prefix = strings.TrimPrefix(filepath.ToSlash(absRoot), filepath.ToSlash(toplevel)+"/")`. The trailing `"/"` is always a forward slash because both operands are post-`ToSlash`." Or: add a Note "rak targets Unix only; Windows support is v0.2+" and a `//go:build !windows` guard on `internal/lister/git.go`.

### Counterexample C4 — F18 "hidden check on each path segment" but `fileset.IsHidden` takes a single name, not a path

- **Severity:** blocker
- **Attack target:** F18 — "(a) hidden check via `fileset.IsHidden` on each path segment of `relPath`"
- **Construction:** `fileset.IsHidden(name string) bool` (verified at `internal/fileset/file.go:117`) accepts a single basename and returns true if it starts with `.` (excluding `.` and `..`). It does NOT split paths. So `fileset.IsHidden("sub/.hidden/file.txt")` returns **false** — the string `"sub/.hidden/file.txt"` does not start with `.`. The plan claims hidden-check applies "to each path segment" but does not specify the split. Unit 4.2's acceptance ("hidden (check each segment of `relPath` via `fileset.IsHidden`)") is similarly vague. The builder must do `strings.Split(relPath, "/")` and call `IsHidden` per segment — but that's not in the F18 wording. A different builder will write `IsHidden(relPath)` (single call, wrong) and pass a test that only covers root-level files. The unit-4.2 test `TestGitLister_FilterHidden` only tests `.gitignore` at root (a single-segment path), so the per-segment bug is not caught.
- **Mitigation status:** UNMITIGATED — plan must change.
- **Suggested fix:** Reword F18 to: "Hidden check: split `relPath` on `/`; if any resulting segment satisfies `fileset.IsHidden(segment)` and `!opts.IncludeHidden`, skip the file." Add a `TestGitLister_FilterHidden_Nested` subtest in Unit 4.2 acceptance: `.config/secrets/credentials.txt` (which git tracks) must be hidden-filtered.

### Counterexample C5 — F19 `--no-gitignore` no-op violates user-trust contract and contradicts decision 10

- **Severity:** blocker
- **Attack target:** F19 + Notes § "--no-gitignore Flag Open Issue" + UN1
- **Construction:** Decision 10 enumerates `--no-gitignore` as an "escape hatch" to "respect `.gitignore`" by default. A user types `rak --no-gitignore` expecting to count **untracked** files (the literal meaning of "no gitignore"). F19 says the flag is a silent no-op in git repos. The plan routes this to UN1: "should we document this in `--help`?" That misframes the issue. The problem isn't documentation — it's that **the flag's name now lies about its behavior in 50%+ of the user's invocations** (anyone running rak inside their own project). Decision 32's "default to tracked-only when in a git repo" is correct, but the escape hatch was meant to PRESERVE the Drop 3 behavior. The correct fix is: `--no-gitignore` falls back to `WalkLister` even when in a git repo. That keeps the flag semantically honest and preserves the user's escape from tracked-only mode. The plan should treat this as a `Detect` policy: `Detect(ctx, root, fsys, opts)` checks `opts.DisableGitignore` BEFORE running `git rev-parse` and returns `WalkLister` when the user opted out.
- **Mitigation status:** UNMITIGATED — plan must change.
- **Suggested fix:** Reword `Detect` body in Unit 4.1 acceptance: "If `opts.DisableGitignore == true`, return `WalkLister` directly without running `git rev-parse` — the user explicitly opted out of git-aware behavior. Otherwise run `git rev-parse --is-inside-work-tree`; on exit 0 return `GitLister`, else return `WalkLister`." Drop F19. Update UN1 to "RESOLVED — `--no-gitignore` forces WalkLister even in git repos." This is decision-level; surface in Phase 3 discussion.

### Counterexample C6 — Integration tests run with CWD = `main/cmd/rak` (inside the rak repo); `lister.Detect` returns GitLister and enumerates rak's own tracked files instead of the test fixture

- **Severity:** blocker
- **Attack target:** Unit 4.4 acceptance — "`runRoot`: ... call `lister.Detect(ctx, args[0], os.DirFS(args[0]), listerOpts(flags))`" + Notes § "Integration Test Impact on testdata/tree"
- **Construction:** `cmd/rak/integration_test.go` runs `cmd.SetArgs([]string{"--format=human", "testdata/tree"})` (lines 159–165). `lister.Detect` calls `exec.CommandContext(ctx, "git", "rev-parse", "--is-inside-work-tree")` with `cmd.Dir = filepath.Abs("testdata/tree")` per F16. That resolves to `/Users/.../main/cmd/rak/testdata/tree`. `git rev-parse --is-inside-work-tree` from inside that dir returns "true" (it's nested inside the rak repo). `Detect` constructs `GitLister` with `toplevel = /Users/.../main` (the rak repo root!) and `prefix = "cmd/rak/testdata/tree"`. `GitLister.List` then runs `git ls-files --full-name -z` from `cmd.Dir = toplevel = main/` and filters to `prefix = cmd/rak/testdata/tree`. The result: only the **tracked** files under `cmd/rak/testdata/tree/` are listed — which is what we want — BUT the `fsys` field of `GitLister` was constructed from `os.DirFS(args[0])` = `os.DirFS("testdata/tree")`, which is rooted at `cmd/rak/testdata/tree/`. So when `newFile(g.fsys, relPath, relPath)` is called with `relPath = "a.txt"` (after stripping the `cmd/rak/testdata/tree/` prefix), the file Open works correctly. So far so good. BUT — and this is the real bug — `prefix` computation uses F17's `string(filepath.Separator)` which on macOS works by coincidence, AND the `runTreeFS` helper (which is the test seam) injects an `fstest.MapFS` whose paths are NOT inside the rak repo. The injected `fstest.MapFS` has files like `vendor/ignored.txt` that do not exist on disk. `Detect` will run `git rev-parse` against `os.DirFS` ... wait — `Detect` uses the actual absolute path of `root`, not `fsys`. So for `runTreeFS` with an `fstest.MapFS`, `root` is `"."` and `filepath.Abs(".")` = `main/cmd/rak` (the test's CWD). `Detect` then returns `GitLister` for `main/cmd/rak`. `GitLister.List` shells `git ls-files` and yields rak's own source files (`root.go`, `root_test.go`, etc.), NOT the injected MapFS files. **Every single `TestRootCmd_PathArg_*` test breaks.** This is C2's deeper manifestation: `runTreeFS` MUST bypass `Detect`.
- **Mitigation status:** UNMITIGATED — plan must change.
- **Suggested fix:** Same as C2 — pin `runTreeFS` to construct `lister.NewWalkLister(fsys, ".", listerOpts(flags))` directly, never call `Detect`. Add a hard-pin note in Unit 4.4 acceptance: "`runDirectory` accepts a `FileLister`. Production code (`runRoot`) wraps `lister.Detect`. Test code (`runTreeFS`) constructs a `WalkLister` directly — `Detect` MUST NOT be called from a test that injects `fstest.MapFS`."

### Counterexample C8 — toon-go's behavior on string values containing the chosen delimiter is unverified; F20 is decorative

- **Severity:** major
- **Attack target:** F20 — "Pipe (`|`) is extremely rare in filesystem paths and is a safe alternative."

F20's "pipe is safe" claim assumes toon-go either (a) quotes/escapes values containing pipes, OR (b) panics, OR (c) silently corrupts. The plan presumes (a) without evidence. Counter-construction: a directory path `/Users/foo/projects/api|backup` (rare but legal on POSIX; `|` is a legal filename character) flows through `Directory.Path` into a tabular TOON array row. The decoder is not rak's problem — but a downstream LLM that round-trips the output is. Decision 33's "TOON for LLM consumption" implies decoder fidelity. The plan needs evidence that toon-go escapes embedded delimiters; without it, the F20 "safe alternative" claim is wishful.
- **Mitigation status:** UNMITIGATED — plan must change.
- **Suggested fix:** Add a Unit 4.5 acceptance bullet: "Add a `TestTOONRenderer_RenderTree_PipeInPath` subtest that constructs a `Directory{Path: "api|backup", Counts: ...}` and verifies the round-trip via `toon.Unmarshal` returns the original path string. If toon-go does not escape embedded pipes, fall back to a different strategy (JSON-escape paths before marshaling, or use a non-printable delimiter)." If the round-trip fails, escalate to dev — this is a decision-33 contract violation.

### Counterexample C9 — Submodule + sparse-checkout + worktree edge cases unaddressed; `main/` is itself a worktree

- **Severity:** major
- **Attack target:** Unit 4.1 acceptance + Unit 4.2 acceptance + F16
- **Construction:** Multiple edge cases the plan does not specify:
  1. **Worktree (the dev's actual setup).** `cat /Users/.../main/.git` returns `gitdir: /Users/.../worktrees/main` — `main/` is a worktree, not the primary checkout. `git rev-parse --is-inside-work-tree` from `main/` returns "true"; `git rev-parse --show-toplevel` returns `/Users/.../main` (the worktree root, not the bare repo). `git ls-files` works correctly. This is fine, but the plan should call it out — Drop 4 development happens FROM a worktree.
  2. **Submodule.** If the user runs `rak path/to/submodule`, `git rev-parse --show-toplevel` returns the submodule's toplevel, not the super-project's. `git ls-files --full-name` then lists submodule-relative paths. The plan doesn't specify whether submodules are walked as separate units or rolled into the parent. Today's `fileset.Walker` would walk submodule contents as ordinary directories (no special handling). After 4.4, the behavior diverges — `GitLister` only lists the super-project's tracked files (submodule contents are NOT tracked by the super-project's `git ls-files`), so a user invoking `rak .` in a repo with submodules loses all submodule content. This may be desired or undesired; the plan should pick one.
  3. **Sparse-checkout.** `git ls-files` lists all tracked files; some may not exist in the working tree under sparse-checkout. `newFile(g.fsys, relPath, ...)` succeeds, but `f.Open()` returns `fs.ErrNotExist`. Walker continues (per F6). The plan doesn't pin: is this a per-entry error in the aggregated summary, or should GitLister filter to `git ls-files --stage` (only present files)? `git ls-files` has a `--exclude-from` / `--cached` distinction; the plan picks `--full-name -z` which lists all index entries.
- **Mitigation status:** UNMITIGATED — plan must change.
- **Suggested fix:** Add a Notes paragraph "Edge Cases — Submodules, Sparse-Checkout, Worktrees": (a) Submodules: GitLister will NOT recurse into them in v0.1.0; document as a known limitation. (b) Sparse-checkout: `git ls-files` returns index entries; non-present files surface as `f.Open()` errors aggregated into the renderer's error summary (matches F6). (c) Worktrees: explicitly supported; `git rev-parse --show-toplevel` returns the worktree root which is the correct enumeration scope.

### Counterexample C10 — F17 empty-prefix case has an off-by-one when `absRoot == toplevel`

- **Severity:** major
- **Attack target:** F17 + Unit 4.2 acceptance "`prefix = strings.TrimPrefix(root, toplevel+"/")` (empty when `root == toplevel`)"
- **Construction:** When `absRoot == toplevel` exactly (user runs `rak .` at the repo root), `toplevel + "/"` is `toplevel + "/"` (e.g. `/Users/foo/repo/`). `strings.TrimPrefix(absRoot, toplevel+"/")` = `strings.TrimPrefix("/Users/foo/repo", "/Users/foo/repo/")` = `"/Users/foo/repo"` (NO trim, because the prefix is longer than the input). So `prefix` ends up as the FULL absRoot — not the empty string the plan claims. The downstream filter `strings.HasPrefix(repoRelPath, prefix)` then never matches (`repoRelPath` is relative, `prefix` is absolute). GitLister yields zero files. The plan's "`(empty when root == toplevel)`" parenthetical is wrong. The builder must explicitly check `if absRoot == toplevel { prefix = "" }` before the TrimPrefix.
- **Mitigation status:** UNMITIGATED — plan must change.
- **Suggested fix:** Reword Unit 4.2's `newGitLister` body in acceptance: "If `absRoot == toplevel`, set `prefix = ""`. Otherwise compute `prefix = strings.TrimPrefix(filepath.ToSlash(absRoot), filepath.ToSlash(toplevel)+"/")`. Assert in a unit test that prefix is exactly `""` for the equal case." Combine with C3's normalization fix.

### Counterexample C15 — Plan misses the `--depth` flag's tracked-mode semantic; depth math may differ between GitLister and WalkLister

- **Severity:** minor
- **Attack target:** F18 (b) — "depth check via `strings.Count(relPath, "/")` — skip if depth exceeds `opts.Depth` (0 = no limit)"
- **Construction:** Walker's depth math (verified at `walker.go:144,217`): `rootDepth = slashCount(w.root)`; per-entry depth `= slashCount(p) - rootDepth`. The walk root is at depth 0. **`opts.Depth = 1` means "walk the root only, no subdirectories"** (per `walker.go:227`). GitLister's depth math per F18: `strings.Count(relPath, "/")`. `relPath` is the path RELATIVE to the walk root (per F17). So `relPath = "a.txt"` has 0 slashes (depth 0); `relPath = "sub/nested.txt"` has 1 slash (depth 1). The walker's `Depth = 1` means SKIP depth 1 files. GitLister's `strings.Count(relPath, "/") >= opts.Depth` would also skip `sub/nested.txt` (1 >= 1). **Same result for this case.** BUT — Walker uses `depth >= w.opts.Depth` (per walker.go:222 + 227). GitLister per F18 uses "skip if depth exceeds `opts.Depth`" — "exceeds" reads as `>` not `>=`. If the builder writes `if depth > opts.Depth { skip }`, then `Depth=1` includes depth-1 files in GitLister but excludes them in Walker. Inconsistent semantic between listers, off-by-one bug latent in F18's "exceeds" wording.
- **Mitigation status:** UNMITIGATED — plan must change.
- **Suggested fix:** Reword F18 (b) to: "depth check via `strings.Count(relPath, "/") >= opts.Depth && opts.Depth != 0` — skip when the count is >= the limit, matching Walker's `depth >= w.opts.Depth` rule at `walker.go:227`. `opts.Depth == 0` disables the limit."
