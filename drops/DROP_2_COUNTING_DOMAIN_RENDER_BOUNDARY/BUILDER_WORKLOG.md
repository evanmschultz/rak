- **Design notes:**
  - **Factory pattern preserved, flag var lives in the closure.** Drop 1's scaffold already had `newRootCmd()` as a factory — good, because test hygiene depends on it. The `format` variable lives *inside* `newRootCmd` rather than at package scope. Package-level `var format string` would alias across every test case since cobra `StringVarP` writes through the pointer, and subtests running in parallel would race on that shared state. Per-factory-call binding is the idiomatic Go-CLI testing shape; it also removes any need to reset global state between tests.
  - **`selectRenderer` as a small pure helper.** Extracted so the mapping `format string → Renderer` is one switch statement, not buried inside `runRoot`. Makes the format-validation error message (`invalid --format %q: want auto | human | json`) the single source of truth and keeps `runRoot` to a linear error-propagation chain.
  - **`"auto"` == `"human"` explicit fallthrough, no TTY inspection at the CLI layer.** Per PLAN.md acceptance bullet 4 and the design notes in Unit 2.2, laslig's `NewHumanRenderer` constructs its printer per `Render` call against the passed writer, so `ResolveMode` runs against the real `cmd.OutOrStdout()` — TTY when the binary runs interactively, plain when stdout is a pipe or a `bytes.Buffer`. rak's CLI doesn't need its own `isatty` check; treating `"auto"` and `"human"` as the same case keeps the switch minimal.
  - **`cobra.MaximumNArgs(1)` + in-RunE `len(args)==1` reject.** The cobra-level `MaximumNArgs(1)` guards against `rak foo bar` without touching RunE — cobra errors with its own usage message. The `len(args)==1` path is handled inside `runRoot` so the error message can mention Drop 3 explicitly (F1 acceptance bullet) rather than using cobra's generic "too few/many arguments" phrasing. This keeps the UX hint precise without losing the framework-level cap on `args>1`.
  - **`_ = c.Context()` retained.** The Drop-1 stub already threaded this; Drop 2's wiring keeps it for forward-compatibility. Drop 3's walker will check `ctx.Done()` in its traversal loop, and Drop 8's parallel walk will use it to cancel worker goroutines. No-op today, cheap insurance against a future refactor that forgets to re-thread it.
  - **Assert labels, not exact bytes.** The stdin/human default test asserts `Bytes`/`Lines`/`Words`/`Chars` appear in the buffer rather than pinning an exact laslig-formatted byte sequence. Exact-snapshot coverage already lives in `internal/render/render_test.go` via `newHumanRendererWithMode` — duplicating it here would double-pin the laslig layout for no added signal and couple `cmd/rak` tests to laslig's (non-pinned-in-production-mode) internal formatting policy. The CLI-layer test's job is to prove the wiring picks the right renderer and plumbs the writer through; it delegates "does the output look right" to the render package.
  - **JSON test asserts quoted keys, not exact bytes.** Same reasoning — the render package snapshots the exact JSON. The CLI test only needs to prove `--format=json` picked `NewJSONRenderer` and the keys reached stdout.
  - **staticcheck QF1011 as a pin against premature type annotations.** The initial draft had `var reader io.Reader = c.InOrStdin()` to make the input type explicit at the call site. staticcheck flagged it — `cmd.InOrStdin() io.Reader` already returns the interface type, so the declaration was pure noise. Removing it dropped the `"io"` import too. Net shape is smaller and more idiomatic. Worth noting for future units: any `var x T = expr` declaration where the RHS already has type `T` will trip QF1011; use `x := expr` and let inference do its job.

Alternatives considered: `Record` (has `Fields` but renders label with trailing `:` per `printer.go:394-400` — wrong shape for bytes/lines/words/chars columnar output), `Table` (over-kill for 4 scalar rows), `StatusLine` (wrong semantic). KV is the right fit: aligned label-value pairs, no title needed (left empty — KV renders without a heading when `Title == ""` per `printer.go:181`).
  - **Per-call printer construction inside `Render`:** the laslig.New / laslig.NewWithMode printer is constructed per `Render` call bound to the passed writer, not stored on the `humanRenderer` struct. Per-call construction is cheap: `newPrinter` is just a struct alloc (`printer.go:54-66`).
  - **Test-mode values captured as a single `var testHumanMode`:** DRYs the mode literal across five call sites (snapshot test + three subtests + the second snapshot test), and makes the F3 pin visible and auditable in one place. The comment on the var cites the F3 invariant explicitly.

Observed shape: leading `\n` (laslig's `leadingGap: 1` for the first content block per `DefaultLayout()` at `policy.go:158-167` + `beginBlock` at `printer.go:335-346`), `"  Bytes  0\n  Lines  0\n  Words  0\n  Chars  0"` body (two-space indent + 5-char label + two spaces + `%-*s` right-padded value + newline per pair per `printer.go:206-217`), trailing `\n` from `writeContentString`'s `fmt.Fprintln` at `printer.go:572`. All four labels are 5 chars so width is 5 uniformly — no varying label padding.
  - **JSON renderer trailing newline:** stdlib `json.Encoder.Encode` appends `\n` — confirmed in Go docs and the resulting snapshot `{"Bytes":12,"Lines":1,"Words":2,"Chars":12}\n`. Field order matches `counting.Counts` declaration order (Bytes, Lines, Words, Chars) because the struct carries no `json:` tags (F4 pin).
  - **No context.Context in Render:** the `Renderer` interface signature is `Render(w io.Writer, counts counting.Counts) error`. `cmd.Context()` lives at the cobra layer (Unit 2.3) and does not need to thread into rendering — no network, no long-running streams, no cancellation-relevant work inside Render. Keeping the signature small preserves YAGNI; Unit 2.3 can still honor ctx before calling Render if it wants.
