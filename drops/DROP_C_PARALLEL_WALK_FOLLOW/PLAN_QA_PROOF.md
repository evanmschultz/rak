## 1. Premises Verified

- 1.1 [Axis: spec-conformance] `golang.org/x/sync v0.20.0` IS present in `go.mod` line 47 as `// indirect`. Verified directly. C.1 promotion task is well-formed.
- 1.2 [Axis: spec-conformance] errgroup API claims confirmed via Context7 (`/websites/pkg_go_dev_golang_org_x_sync`):
  - `WithContext(ctx)` returns derived context cancelled on first non-nil `Go` error OR first `Wait` return.
  - `Group.Go(f)` blocks when the goroutine count would exceed the configured limit.
  - `Group.SetLimit(n)` — n=0 prevents any goroutines; negative = no limit; "must not be modified while any goroutines are active" (the plan's `if n < 1 { n = 1 }` clamp correctly avoids n=0 deadlock).
  - `Group.Wait()` returns first non-nil error.
  All four match the plan's design decisions verbatim.
- 1.3 [Axis: spec-conformance] Walker is genuinely a pure iterator today. `Walker.Walk` returns `iter.Seq2[*File, error]`, the only state mutated is the captured-by-closure `roots`/`matcher`/`yieldOK`/`rootDepth` — all serial. There is no per-file I/O inside the walker; `f.Open()`/`f.Peek()`/`f.IsBinary()` are all called by the consumer (currently `walkAndCount` in `cmd/rak/root.go`). Placing the worker pool in `walkAndCount`, not in the walker, is the correct layering call.
- 1.4 [Axis: spec-conformance] Current `walkAndCount` I/O surface is exactly: `f.IsBinary()` (`Peek(512)`), `f.Open()` (twice — once for `lang.Split`, once for `countFile`), `counting.Count(rc)`, `lang.Detect(f)` (a third Peek inside `Detect`), `lang.Split(rc, detectedLang)`. All purely per-file, no shared mutable state, no order dependency. Safe to parallelize.
- 1.5 [Axis: spec-conformance] The plan's accumulation list (`byDir`, `byDirLang`, `byDirFiles`, `totalByLang`, `total`, `aggErrs`, `acceptedFiles`) matches the actual local variables declared in `walkAndCount` (root.go:349–355). Mutex-protected-on-write-only is the correct minimal-critical-section pattern.
- 1.6 [Axis: spec-conformance] Channel-feed pattern (single producer = main, N consumers = workers, buffered cap = workers, producer closes on done, errgroup.Wait drains) is a textbook correct producer-consumer shape — no double-close, no send-on-closed (since only the producer closes), no goroutine leak as long as on early errgroup-context cancel the producer also drains via `ctx.Done()` (plan calls this out in Decision 4).
- 1.7 [Axis: acceptance-criteria-coverage] `--max-files` thread-safety claim is correct: in current `walkAndCount` (root.go:438 — `acceptedFiles++` and root.go:443 — `if maxFiles > 0 && acceptedFiles >= maxFiles`), the increment fires AFTER the per-file work completes. Under the parallel design, if main is the only thread that pushes to the channel AND main reads/increments `acceptedFiles` post-yield (or pre-push), the single-writer invariant holds.
- 1.8 [Axis: acceptance-criteria-coverage] `summary.SortDirs` IS called post-walk (root.go:300, inside `runDirectory`, after `walkAndCount` returns its slice). Plan's order-stability claim — output is deterministic across worker counts — is therefore structurally true.
- 1.9 [Axis: spec-conformance] `fs.WalkDir` symlink semantics confirmed via Go stdlib docs (and `walker.go:114` comment: "fs.WalkDir does not follow them (stdlib-documented behavior)"). The plan's claim that follow requires switching to `filepath.WalkDir` on real OS paths is correct.
- 1.10 [Axis: spec-conformance] `syscall.Stat_t.Dev` + `Ino` exist on darwin and linux (Context7 confirmation). On darwin `Dev` is `int32` (per the Darwin amd64 Stat_t doc) and `Ino` is `uint64`; on linux both are integral. The plan's `uint64(s.Dev)` and `uint64(s.Ino)` casts compile cleanly on both. Windows fallback to path-based dedup is acceptable.
- 1.11 [Axis: spec-conformance] All 14 inbound callers of `fileset.NewWalker` enumerated via Hylla: 12 test functions in `internal/fileset/walker_test.go`, plus `lister.newWalkLister` and `lister.NewWalkLister`. The plan's "fourth-arg defaults to empty string" pattern is backward-compatible (all current callers can be updated to pass `""`, or the constructor can take a variadic option / a sibling factory). Spec is build-friendly.
- 1.12 [Axis: spec-conformance] `--follow` is silent no-op under `GitLister` is correct: `GitLister.List` (`internal/lister/git.go:126`) never consults `g.opts.Follow`; it iterates `git ls-files` output. Git's own enumeration includes symlink-reachable paths that are tracked, so the user-visible behavior is "follow had no effect because git already had the files." Documenting in the `WalkOptions.Follow` doc comment is sufficient.

## 2. Findings

- 2.1 [Axis: spec-conformance] [severity: medium] **Plan misses `SingleFileLister` as a third concrete FileLister.** `internal/lister/single.go` defines `SingleFileLister` (yields exactly one file) and `Detect` returns it for regular-file roots (lister.go:69–73). The plan's narrative for C.5 mentions only `GitLister` + `WalkLister`. `--follow` on a single-file root is also a no-op (no walking to do) and the worker pool from C.2 is wasteful but harmless on a 1-file workload. Update C.5's Notes (or C.4's "no-op under GitLister" paragraph) to also mention `SingleFileLister`. Builder fix: add a sentence to the `--follow` doc string and one test case `TestRunDirectory_FollowOnSingleFile` showing no error.
- 2.2 [Axis: spec-conformance] [severity: medium] **C.4 doc string for `WalkOptions.Follow` is unspecified.** The plan says to document "the no-op under GitLister" in `WalkOptions.Follow`'s doc comment but does not state the actual doc-comment text or any cycle-detection semantics. The current `WalkOptions` doc style (walker.go:16–59) is exhaustive — every field has a 2–6 sentence description covering defaults, semantics, and cross-field interactions. Add an explicit acceptance bullet to C.4: "`WalkOptions.Follow` carries a doc comment describing (a) default false = stdlib `fs.WalkDir` behavior, (b) true requires `rootPath != ""` and uses `filepath.WalkDir`, (c) cycle detection via dev+inode on unix and path-dedup on windows, (d) `--follow` is no-op under `GitLister` and `SingleFileLister`."
- 2.3 [Axis: spec-conformance] [severity: medium] **C.4's `filepath.WalkDir` branch must replicate every existing walker filter.** Today's `Walker.Walk` does hidden-skip, depth enforcement, per-directory gitignore ingestion, include/exclude matcher, F14 yield-false guard, and root.go-style error wrapping. The plan only specifies the cycle-detection delta — it does NOT call out that the new `filepath.WalkDir`-based code path must continue to apply ALL those filters or that the implementation should factor them so both branches share a single filter helper. Without this, the builder could ship a follow-branch that mis-applies hidden/depth/gitignore and pass narrow follow-only tests while silently regressing on the unified behavior. Add explicit acceptance to C.4: "`Follow: true` walks apply identical hidden/depth/gitignore/include/exclude/yield-false semantics as `Follow: false` walks — verified by a parallel table of tests that run each `WalkOptions` permutation under both modes against a real-OS fixture."
- 2.4 [Axis: acceptance-criteria-coverage] [severity: medium] **C.2 missing a `--max-files` parallel-correctness test.** The plan asserts "main goroutine is sole writer of `acceptedFiles`" but provides NO acceptance test that exercises `--max-files` under workers > 1 with a fixture where racy increment-by-workers would over- or under-count. Add `TestWalkAndCount_MaxFilesUnderWorkers` (workers=4 or 8, fixture with say 50 files, `--max-files=10`): assert the abort fires at exactly 10 accepted files, and that the error wraps `ErrMaxFilesExceeded`. Without this, the design decision is asserted but not verified.
- 2.5 [Axis: acceptance-criteria-coverage] [severity: medium] **C.2 missing an error-aggregation idempotency test.** `walkAndCount` mutates `aggErrs` from multiple worker goroutines under the proposed design (split errors, count errors, binary-check errors all append to `aggErrs`). The plan requires mutex-on-accumulation generally but never states that `aggErrs` is part of the mutex-protected set, and there's no test that exercises a fixture where N>=2 files all produce errors and asserts the final `aggErrs` slice length and contents are exactly the expected set. Add `TestWalkAndCount_AggErrsMutexProtected` (fixture: 10 files with deliberately induced binary-check errors, workers=8) asserting len(aggErrs)==10 and no duplicate-append pattern. Without this, the race detector might still pass on small workloads but real users hit data corruption.
- 2.6 [Axis: acceptance-criteria-coverage] [severity: medium] **C.2 missing a context-cancellation propagation test.** Plan Decision 4 claims "first worker error cancels the derived context, which propagates to workers via I/O paths and to the iterator via `ctx.Done()`." There is no acceptance test that constructs a fixture where one worker returns an error (e.g. an injected I/O failure) and verifies (a) the call returns the wrapped error, (b) remaining workers exit promptly rather than continuing to drain the channel, (c) the producer (main goroutine iterating `source.List(ctx)`) stops pushing. Add `TestWalkAndCount_FirstErrorCancels` with an `fstest.MapFS` whose `Open` returns an injected error after N reads. Without this, the cancellation chain is asserted but unverified.
- 2.7 [Axis: spec-conformance] [severity: low] **`fstest.MapFS` injected-error pattern needs concrete recipe.** Tests like `TestWalkAndCount_FirstErrorCancels` and `TestWalkAndCount_AggErrsMutexProtected` need a way to make `f.Open()` or `f.IsBinary()` fail deterministically for specific files. `fstest.MapFS` does not natively support per-file injected errors. Builder should either (a) use a wrapper `fs.FS` that intercepts `Open`, or (b) use real `t.TempDir` with permission tweaks (chmod 0000 on darwin/linux). Plan should pick one approach so the builder doesn't pick the other in error and fail in CI. Recommend (a) — wrapper FS — for portability.
- 2.8 [Axis: parallelization-graph] [severity: low] **C.4 `blocked_by: —` is correct but worth double-checking with cmd-side `--follow` plumbing.** C.4 (internal/fileset) is parallel-safe with C.1, C.2, C.3 (cmd/rak). C.5 correctly blocks on both C.3 and C.4. C.6 correctly blocks on C.5. Graph is well-formed; no cycles. One nuance: if the orchestrator wants to dispatch C.4 and C.1 concurrently, they truly are disjoint by paths (`internal/fileset/` vs `go.mod`/`go.sum`). Aggressive parallelism per memory `feedback_parallelize_aggressively.md` is achievable.
- 2.9 [Axis: spec-conformance] [severity: low] **Build-tag filenames `follow_unix.go` / `follow_windows.go` need a guard.** The plan shows `//go:build !windows` for `follow_unix.go`. Go's modern build-tag convention uses `//go:build` (1.17+) AND file-name suffixes (`_linux.go`, `_darwin.go`, `_windows.go`). The filename `follow_unix.go` is not a recognized GOOS suffix, so the file is included on every platform unless gated by the `//go:build !windows` tag. The plan does include the tag, but it's worth making explicit in the acceptance: "`mage build` and `mage ci` pass on both darwin and linux locally and on CI's matrix." If CI matrix is single-platform, this isn't proven until a Windows user trips on it. Acceptance is fine as written but flag for the builder.
- 2.10 [Axis: spec-conformance] [severity: low] **C.2 channel-buffer sizing should be specified more precisely.** Plan says "Channel buffer size = `workers`". That's reasonable but uncommitted. With `cap=workers` the producer can run one round ahead of the slowest worker; with `cap=2*workers` the iterator can stay further ahead and smooth out per-file I/O variance. The plan should pin one (recommended: `cap=workers` — minimal memory, errgroup back-pressure does the rest). Builder fix: add a one-line decision in C.2 design notes.
- 2.11 [Axis: spec-conformance] [severity: low] **`errgroup.Wait()` already waits for all goroutines — `close(channel)` ordering is subtle.** Plan says "close the channel; errgroup.Wait() collects any worker error." This is correct ONLY IF the close happens after the producer's final push AND before `eg.Wait()`. The conventional pattern is: producer goroutine does `defer close(ch); for ... { ch <- v }`, while consumer goroutines do `for v := range ch { ... }` (so they exit when the channel closes), and the main thread calls `eg.Wait()` after the producer-spawning Go call. Plan's wording leaves the close-then-Wait ordering implicit. Builder fix: spell out the close-then-Wait pattern in C.2 design notes ("workers `range` the channel; producer `defer close(ch)`; main thread calls `eg.Wait()` after iteration").
- 2.12 [Axis: acceptance-criteria-coverage] [severity: low] **C.6 feature trio for `--workers` lacks a concrete fixture path.** The plan says "use `main/` itself or a real-world fixture large enough for the gif to be honest." `main/` at v0.1.4 may be too small for a visible parallel speedup; the gif could end up showing no measurable difference (which would be a counter-marketing problem). Builder fix: pick a known-large public fixture (e.g. the Linux kernel checkout via shallow clone, or the Go stdlib) and pin it in the tape. Or accept that the gif is illustrative and add a note in the README narrative.
- 2.13 [Axis: spec-conformance] [severity: low] **C.5 `--follow` flag usage string says "macOS + Linux only in v0.2.0" but Windows fallback is path-based.** The plan elsewhere says Windows falls back to path-based dedup and "catches most cycles but not hard-link loops." That's a working-on-windows-with-degraded-guarantees story, not a not-supported-on-windows story. The flag usage string text is misleading. Builder fix: rephrase to "follow symbolic links during walk; cycle detection (dev+inode on macOS/Linux, path-based on Windows)." Or pick one story and commit (either Windows-not-supported-at-all and gate the flag with build tags, or Windows-supported-with-caveats).
